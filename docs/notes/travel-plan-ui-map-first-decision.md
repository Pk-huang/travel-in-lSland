# Travel Plan UI / Map First Decision

日期：2026-07-24
狀態：accepted

## 背景

目前已完成旅行計劃資料契約與全日期靜態資料（8/26 ~ 9/5），但使用者最優先的可見需求集中在：
- 地圖圖標顯示行程順序
- 左側單日行程可直接導引 Google Maps
- 左側行程清單的切換與可編輯能力
- 地圖顯示使用者當前位置

為了先建立可操作、可演示、可驗收的核心體驗，決議本輪先做 UI 與地圖互動閉環。

## 決策

本輪採「UI / 地圖先行」策略，執行順序如下：
1. 地圖站點標記顯示第幾站（行程順序）。
2. 單日行程點擊導引 Google Maps（單站）。
3. 左側切換全部行程 / 單日行程。
4. 單日行程編輯（拖曳、新增、刪除、欄位編輯）。
5. 全部行程編輯（跨日拖曳、新增、刪除）。
6. 使用者定位顯示（權限導向）。

## Google Maps URL 規則

### 單站導引
- 優先使用 `lat/lon`：
  - `https://www.google.com/maps/dir/?api=1&destination={lat},{lon}&travelmode=driving`
- 無座標時 fallback 地名：
  - `https://www.google.com/maps/dir/?api=1&destination={encodedLabel}&travelmode=driving`

### 單日路線（後續步驟）
- 由當日 stops 生成：`origin + destination + waypoints`。
- waypoint 過多時需裁切，避免 URL 過長。

## 保留方案（不納入本輪）

- 激進實驗：自動排程建議（依天數、開車時數、景點類型自動重排）。
- 保留原因：
  1. 依賴規則校正與資料完整度，變數高。
  2. 先穩定基礎 UI/地圖互動，再導入智慧重排可降低除錯成本。

## 風險與對策

1. 定位權限被拒絕：
- 對策：顯示清楚提示，保留手動導引功能。

2. 站點缺座標導致導引精度下降：
- 對策：以地名 fallback，不阻斷流程；後續補齊高頻站點座標。

3. 拖曳編輯造成資料不一致：
- 對策：先做 schema 驗證與最小防呆，再擴充 undo/版本快照。

## 驗收方式

1. 點左側單日任一站點，可開啟 Google Maps 導引。
2. 地圖可明確辨識站點順序。
3. 可切換全部行程 / 單日行程並保持顯示一致。
4. 單日編輯流程可完成新增、刪除、拖曳與欄位修改。
5. 使用者授權定位後，可在地圖看到當前位置。
