# 動態 LOD 地形設計筆記（面試導向）

> 目的：把「放大時被放大的範圍能更清晰」這個需求，用 LOD（Level of Detail）解。
> 本專案為**面試作品**，重點是「展示懂 LOD 的設計思維 + 有會動的 demo + 能講清楚取捨」，
> 而非做出 production 級的全球地形引擎。
> 目前狀態：已進入實作與策略收斂階段（本文件同時作為規劃與執行中決策紀錄）。

## 2026-07-16 策略轉向：Spot LOD（景點焦點式）

### 轉向原因

1. 完整自由漫遊 LOD 的工程成本與風險過高（接縫、快取、動態載入治理）。
2. 專案目標偏向「可展示、可觀測、可壓測」，非 production 級全球地形引擎。
3. 後續需求已明確：使用者主要透過景點點擊檢視，而非完全自由探索。

### 新目標（目前主線）

1. 提供景點 UI 可點擊。
2. 點擊後相機飛到固定檢視畫面（固定 target / distance / angle）。
3. 僅焦點景點啟用局部細化。
4. 未被點擊景點不啟動 LOD。

### 核心規則

1. `No focus, no LOD.`
2. `Only active POI gets detail.`
3. 轉場與細化流程採 ready-swap，避免切換過程視覺閃斷。

### 與原始 LOD 理想的關係

1. 保留 LOD 核心思想：區域細化、事件/距離觸發、資料就緒後切換。
2. 放棄全域自由漫遊的完整度，換取可交付性與穩定度。
3. 屬於「高可行近似解」，不是完整 quadtree LOD。

### 與壓力測試並行規劃

1. 以景點點擊流程作為固定壓測腳本。
2. 觀測指標：FPS、主執行緒尖峰、切換耗時、記憶體變化、重建次數。
3. 先拿基線，再決定是否增加局部 DEM 高解析 patch。

## 0. 為什麼是 LOD（根因回顧）

- 現況：地形是**單一固定 128×128 網格**，冰島東西 ~527km → 每格 ~4.1km。
- 放大時看到的還是同一張粗網格，只是被拉大 → **塊狀**。這是**解析度問題，不是相機問題**。
- 「無限放大都清晰」的正解＝**LOD**：離得遠用粗的、湊近才換細的，且只換「正在看的那一塊」，
  所以**頂點數恆定**卻能無限深入細節（Google Earth / Google Maps 的原理）。

### 核心觀念兩件事
1. **Tiling / Quadtree（四分樹切塊）**：整塊地切成方格，每格能再切 4 塊，遞迴加深。
2. **LOD selection（依距離選層）**：每幀依「離相機多遠 / 佔螢幕多大」決定每塊用哪種精細度。

## 1. Production 完整版的五個子系統（最難）

| 子系統 | 做什麼 | 估時 |
|---|---|---|
| ① 資料切 tile | quadtree + tile server 動態切、無限深度 | ~3~5 天 |
| ② LOD 選層 | 每個 tile 各自依距離選層級 | ~2~3 天 |
| ③ 視錐 culling | frustum culling 算相機視錐內有哪些 tile | ~1~2 天 |
| ④ 動態載入+快取 | 進視野 fetch、離開釋放、LRU 快取 | ~3~4 天 |
| ⑤ 接縫縫合 stitching | 相鄰不同層級交界防裂縫（最惱人）| ~2~4 天 |
| — 整合/除錯緩衝 | | ~3~5 天 |

**總計 ~2~4 週。** 或不自造輪子，改用 **3D Tiles（loaders.gl）** 或 **Cesium quantized-mesh terrain**，
導入約 1~2 週，但受既有 R3F 架構整合成本影響。

## 2. 方案光譜（最難 → 最簡單，含估時）

### 方案 D：Production 完整版（最難，~2~4 週）
- 上面五個子系統全做，或接 3D Tiles / Cesium。
- 面試 CP 值低：投入巨大、對「固定小範圍的冰島」是過度工程。

### 方案 B：Chunked LOD MVP（真 tiling + 全域單層級，~2.5~3 天）★推薦
- **保留 LOD 靈魂的兩項**：真的切 tile（quadtree-lite）+ 依相機距離選層。
- **砍掉三項包袱**：
  - 全島同一時間用**單一全域層級** → ③視錐 culling 不需要（tile 少、全畫不卡）。
  - 全域單層級 → ⑤接縫**天然消失**（大家同密度，交界不裂）；skirt 僅當保險。
  - 檔案總量 <1MB → ④只需簡單 Map 懶載快取，**不需 LRU 釋放**。
- 資料結構（3 層固定，每 tile 64×64）：

  | 層級 | tile 數 | 每 tile 網格 | 等效全島 | 間距 |
  |---|---|---|---|---|
  | L0（遠）| 1 | 64×64 | 64 | ~8km |
  | L1（中）| 2×2=4 | 64×64 | 128 | ~4km |
  | L2（近）| 4×4=16 | 64×64 | 256 | ~2km |

  - API 預算：L0+L1+L2 = 21 tiles × 4096 = 86,016 點 ≈ 860 req → **一天內抓完**（配額 1000/日）。
- 檔案佈局（離線預切，全靜態）：
  ```
  public/dem/tiles/
    L0/0_0.json
    L1/0_0.json 0_1.json 1_0.json 1_1.json
    L2/0_0.json … 3_3.json  （16 個）
  ```
  產法：擴充現成 scripts/fetch-dem.mjs 加 `--tiles` 模式，跑一次生成全部。
- 元件結構（R3F）：
  ```
  <TerrainLOD>              ← 取代目前 <Terrain>
    ├─ useFrame 讀相機距離 → 算 currentLevel（含遲滯防抖 hysteresis）
    ├─ 依 currentLevel 決定要哪些 tile → 觸發載入
    └─ map 出 <TerrainTile heightmap bbox />   ← 每個 tile 一個 mesh
  <TileCache>              ← Map<string,HeightmapData> + 懶載 fetch
  ```
  關鍵：把現在 Terrain 的「heightmap → geometry」邏輯抽成 `buildTileGeometry(heightmap, bbox)`，
  tile 與單圖共用（順勢還掉「抽 util」技術債）。
- 演算法三動作：
  1. **選層（每幀）**：`distance = camera 到島中心` → 門檻表映射 level；升/降門檻錯開（遲滯）防抖動。
  2. **載入+快取**：切層時該層 tile 若快取沒有就 fetch，存 Map；不釋放。
  3. **無縫**：全域單層級 → 相鄰 tile 同密度、共用邊界取樣 → 天然無接縫。
- 估時：資料切 tile ~1 天、TerrainLOD+TerrainTile 元件 ~1 天、選層+快取+抽 util ~0.5~1 天。
- **可選加深**：加 L3（8×8=64 tile、等效 512、~1km），拓資料要跨 ~2 天，清晰度再升一級。

### 方案 C：偽 LOD / 解析度切換（~1 天）
- **不切 tile**，準備 3 張不同解析度的**整島** heightmap（128 / 256 / 512），依相機距離換整張。
- 最省，但難自圓其說是「LOD」，比較像「解析度切換」，面試講起來力道弱。

### 方案 A：單一固定網格提高解析度（~0.5~1 天，非 LOD 基準線）
- 重跑 fetch-dem 拉到 256×256（間距 ~2km、~270KB）或 512×512（~1MB、API 跨 ~3 天）。
- 不是 LOD，只是整體變細；作為「不做 LOD 時的最省替代」。

## 3. 切層視覺體驗（兩種，之後挑）

| 選項 | 說明 | 額外工時 |
|---|---|---|
| 瞬間切換 | 相機到門檻直接換層，會看到一次「變清晰」跳變 | 0 |
| 短暫淡入 crossfade | 換層時舊層淡出/新層淡入，過場較柔順 | ~0.5 天 |

> 註：MVP 是「**分層切換**」不是「連續 streaming」，切層瞬間本質上會有一次跳變。
> 面試時講清楚這是刻意的 scope 取捨，不是缺陷。
